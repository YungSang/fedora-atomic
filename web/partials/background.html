Packages, Trees, and Images

Packages

Traditional GNU/Linux distributions come as a collection of packages - this is like a grocery store. There's an effectively infinite variety of ways one can assemble the parts. Not all combinations make sense (in fact, very few do), but hopefully none will be actively harmful.

Each time you go to a grocery store (upgrade), you are responsible for reassembling from parts any meals you like.

OSTree Repository

An OSTree repository is more like a restaurant. You can choose from a number of pre-assembled meals (filesystem trees). But not an infinite variety. Each meal has a name, and when you re-visit the restaurant, you can get a newer (and hopefully improved) version of that same meal, or at any time try a different one.

For Fedora and other operating system vendors, OSTree can act like a restaurant in partnership with a grocery store. If you like some of the meals, you can take the ingredients and recipies (packages and scripts), and make your own.

Images

The analogy isn't as strong here, but traditional image-based systems such as ChromeOS and CoreOS are a bit more like a short (or cheap) airplane flight where you have exactly one choice of meal.

Image systems are good in that they fulfill a reliable, targeted purpose. Additionally of course, many of these kinds of operating systems come with application mechanisms on top. For ChromeOS, that's web pages. For CoreOS, it's Docker.

This lets them do quite a bit - but not everything fits cleanly into the "application" or "OS" box. For example, development headers and tools. Neither ChromeOS nor CoreOS have an answer for a mechanism to help you develop the system itself.

Short Term Goal - Testing Rawhide

The Atomic Initiative will track Fedora Rawhide, assembling packages into filesystem trees, and offering them for atomic replication by both human and machine consumers.

Why machine consumers? The very same rpm-ostree codebase actually contains qemu-kvm based testing infrastructure. It's designed to rapidly boot a new updated tree as a virtual guest, running arbitrary testing code, and pulling out telemetry such as log files for display.

Concretely, the build-status results are live - the screenshots will change as the packages change, automatically.

Short Term Goal - Testing OSTree

While the core OSTree codebase is now fairly stable, rpm-ostree is undergoing rapid evolution. It's only recently that SELinux is fully enforcing, for example. The goal is for everything to be packaged, and to become a more integrated part of the Fedora Infrastructure.
