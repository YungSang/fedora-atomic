Compose, then replicate

Anywhere you use RPM packages today to deploy to client systems, the rpm-ostree model allows you to optionally pre-compose that package set on a server. Client systems then have an immutable OS, but with online and atomic upgrades to newer versions.

This OSTree model sits somewhere between traditional package systems and image-based updates. For example, an OSTree "tree" can be deployed on any Unix filesystem and block storage layout. Client systems can pull it, and incrementally download updates online.

In this model, the RPM database lives in /usr/share/rpm, and a compatibility symlink from /var/lib/rpm allows rpm to still operate in read-only mode. You can rpm -q and determine what packages are installed.

Short Term Goal - Testing Rawhide

The Atomic Initiative will track Fedora Rawhide, assembling packages into filesystem trees, and offering them for replication by both human and machine consumers.

Why machine consumers? The very same rpm-ostree codebase actually contains qemu-kvm based testing infrastructure. It's designed to rapidly boot a new updated tree as a virtual guest, running arbitrary testing code, and pulling out telemetry such as log files for display.

Concretely, the build-status results are live - the screenshots will change as the packages change, automatically.

Short Term Goal - Merging RPM-OSTree into Fedora

While the core OSTree codebase is now fairly stable, rpm-ostree is undergoing rapid evolution. The goal is for everything to be packaged, and to become a more integrated part of the Fedora Infrastructure. At the moment, a patched version shadow-utils is the only package not in Fedora.
